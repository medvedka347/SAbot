## 🔍 Выявленные особенности поведения

### 1. Состояние после сохранения
- **Материалы:** После сохранения вызывается `materials_menu()`, которое **переустанавливает** `MaterialStates.menu`
- **События:** Аналогично, `events_menu()` переустанавливает `EventStates.menu`
- **Роли:** После назначения роли состояние **очищается** (`state.clear()`)

### 2. Поведение "Назад"
- Обработчик `back_handler` в `common.py` **всегда** вызывает `state.clear()`
- Это сбрасывает любой FSM-диалог
- Нет накопления истории состояний

### 3. Валидация на каждом шаге
- Каждый state handler проверяет `message.text`
- При отсутствии текста — `return` (молча)
- Ошибки валидации **не меняют** состояние

### 4. Rate Limiting
- FSM не защищает от rate limiting
- При превышении лимита — ответ "Слишком быстро", состояние не меняется
- Нужно учитывать при тестировании

## 🐛 Найденные проблемы (FSM-специфичные)

### FSM-001: Необработанное состояние confirm_announce
**Модуль:** `handlers/events.py`  
**Состояние:** `EventStates.confirm_announce`

При создании события с настроенной `ANNOUNCEMENT_GROUP_ID` бот переходит в состояние `confirm_announce`, но:
- Нет явного handler'а для других ответов (кроме "✅ Да" и "❌ Нет")
- Любой другой текст игнорируется (просто `return`)

**Рекомендация:** Добавить fallback handler или сообщение об ошибке.

---

### FSM-002: Возможная потеря данных при одновременных callback'ах
**Риск:** Низкий

Если пользователь быстро кликает по inline-кнопкам, возможна ситуация:
1. Callback 1: читаем state data
2. Callback 2: читаем state data (те же)
3. Callback 2: обновляем state
4. Callback 1: обновляем state (перезаписываем данные callback 2)

**Митигация:** Rate limiting + блокировка UI после клика.
