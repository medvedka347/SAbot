## 🔴 КРИТИЧЕСКИЕ БАГИ (Требуют немедленного исправления)

### BUG-001: Ошибка в отчёте Льва — показывает не того ментора
**Файл:** `handlers/buddy.py:611`  
**Суть:** В функции `lion_show_mentor_details` используется `callback.from_user.id` (ID Льва) вместо `mentor_id` (ID выбранного ментора).

```python
# НЕПРАВИЛЬНО (сейчас):
mentor = await get_user_by_id(callback.from_user.id)

# ПРАВИЛЬНО (должно быть):
mentor = await get_user_by_id(mentor_id)
```

**Влияние:** 🔴 Высокое — функция отчётов полностью неработоспособна

---

### BUG-002: Race Condition при редактировании
**Файлы:** `handlers/materials.py:271-304`, `handlers/events.py:316-355`  
**Суть:** Два админа могут одновременно редактировать один объект. Второе сохранение перезапишет первое без предупреждения.

**Сценарий:**
1. Админ А выбирает материал ID=5
2. Админ Б выбирает тот же материал  
3. Админ А сохраняет изменения
4. Админ Б сохраняет → перезаписывает изменения А

**Влияние:** 🔴 Высокое — потеря данных

**Решение:** Добавить версионирование (поле `updated_at`) или оптимистичную блокировку

---

### BUG-003: Мультироли ломают доступ
**Файлы:** `handlers/common.py:97-102`, `common.py:130-143`, `common.py:165-173`  
**Суть:** Проверка `role == ROLE_ADMIN` не работает для пользователей с ролью `admin,lion`.

```python
# НЕПРАВИЛЬНО (сейчас):
role = await get_user_role(user_id)
if role == ROLE_ADMIN:  # False для 'admin,lion'

# ПРАВИЛЬНО:
roles = await get_user_roles(user_id)
if ROLE_ADMIN in roles:
```

**Влияние:** 🔴 Высокое — пользователи с мультиролями теряют доступ к функциям

---

### BUG-004: get_all_mentors не видит мультироли
**Файл:** `db_utils.py:1017-1033`  
**Суть:** SQL запрос ищет `role = 'mentor'` (точное совпадение), пропуская `mentor,admin` и `mentor,lion`.

```sql
-- НЕПРАВИЛЬНО:
WHERE role = 'mentor'

-- ПРАВИЛЬНО:
WHERE role LIKE '%mentor%'
```

**Влияние:** 🔴 Высокое — Лев не видит часть менторов в системе

### BUG-009: Дублирующийся update_data в ролях

## 🎨 КЛЮЧЕВЫЕ UX ПРОБЛЕМЫ

### UX-001: Нет подтверждения при удалении ⚠️ КРИТИЧНО
**Где:** Везде (материалы, события, пользователи)  
**Проблема:** Удаление происходит мгновенно одним кликом, без возможности отмены.

**Текущий flow:**
```
🗑️ Удалить → Выбрать раздел → Выбрать материал → [МГНОВЕННОЕ УДАЛЕНИЕ]
```

**Рекомендуемый flow:**
```
🗑️ Удалить → Выбрать раздел → Выбрать материал → 
🗑️ Удалить "Название"?
   Это действие нельзя отменить.
   [✅ Да, удалить] [❌ Отмена]
```

---

### UX-002: Кнопка "Назад" непредсказуема
**Где:** `handlers/common.py:205-264`  
**Проблема:** Поведение разное в разных сценариях:
- В админских диалогах — возвращает на предыдущий шаг
- В публичном просмотре — всегда в главное меню
- При удалении — всегда в главное меню

**Пользователь не может предсказать результат.**

---

### UX-003: Нет кнопки "Отмена" в диалогах
**Где:** Все FSM-диалоги  
**Проблема:** Если пользователь начал добавление и передумал — нет выхода. Только многократное нажатие "Назад" или `/start`.

---

### UX-004: Неинтуитивное редактирование
**Где:** `handlers/materials.py:260-268`, `handlers/events.py:307-313`  
**Проблема:** Формат ввода требует двойной перенос строки:
```
название

ссылка

описание
```

**Проблемы:**
- Не видно текущих значений
- Двойной перенос неочевиден
- Точка для пропуска нестандартна
- Нет валидации по ходу ввода

**Рекомендация:** Пошаговое редактирование с показом текущих значений.

---

### UX-005: Нет предпросмотра перед сохранением
**Где:** Все диалоги добавления  
**Проблема:** Данные сохраняются сразу, без возможности проверить.

## ⚠️ EDGE CASES (Граничные случаи)

1. **29 февраля в невисокосный год** — обрабатывается корректно (возвращает None)
2. **Админ удаляет сам себя** — теряет доступ без восстановления
3. **Смена username во время бана** — можно обойти бан
4. **Дублирование менти** — один менти может быть добавлен несколько раз
5. **Пустой поиск** — возвращает все результаты
